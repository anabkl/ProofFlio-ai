# V3.4 SaaS Design Polish

## Scope

Design and UX polish pass on top of the existing V3.1–V3.3 foundation (landing, auth, evidence
onboarding, persistent editor). No Supabase migrations, RLS/storage policies, auth architecture,
server actions, persistence logic, database schema, GitHub OAuth, CV parsing, AI provider logic,
publishing, payments, or analytics were touched.

## Design audit summary

The V3.3 surfaces were already strong on typography, spacing, and evidence-first copy. The concrete
problems found on inspection at 1440×900, 1280×800, 390×844, and 390×844 FR:

1. **Three different header implementations.** The marketing shell (`AppShell`/`Navigation` in
   `product-views.tsx`), the onboarding flow's own inline header, and the auth pages (which had
   **no header at all** — just empty space above the hero) each looked like separate products
   bolted together instead of one connected app.
2. **The editor was wrapped in the marketing shell**, pulling in the "Plans" pricing link, the
   full marketing CTA, and a marketing footer into what should read as an authenticated workspace.
3. **Mobile header crowding on the marketing nav.** At 390px, the wordmark ("ProofFolio AI"),
   language toggle, "Start" CTA, and hamburger button competed for space; the wordmark visibly
   truncated to "ProofFoli…". The language toggle was also duplicated (shown in the collapsed bar
   *and* again inside the open drawer).
4. **Auth forms were pushed below the fold on mobile.** The proof-point row (`sm:grid-cols-3`,
   which collapses to a single stacked column under 640px) added enough height that the actual
   email/password fields on `/auth/sign-up` required scrolling past three full-width cards before
   they were reachable.
5. No horizontal overflow, broken focus states, or missing `aria-current` were found — the prior
   sprint had already fixed these; verified again after this pass (see Validation below).

## Pages improved

- `/auth/sign-in`, `/auth/sign-up` — now share the same header as the rest of the app; mobile
  proof points compressed into a 3-up compact row so the form is visible without scrolling.
- `/onboarding` — same header content, now sourced from one shared component instead of
  duplicated markup.
- `/editor` — moved off the marketing shell onto the same workspace header as onboarding, with
  the marketing footer removed (a workspace tool doesn't need a marketing footer).
- `/` (landing), `/demo`, `/templates` — mobile header spacing tightened; no other landing changes
  (out of scope per the brief).

## Exact files changed

- `src/components/workspace-header.tsx` — **new.** Single shared header (logo, contextual
  subtitle, Home/Templates/Live demo nav, language toggle, auth status) used by auth, onboarding,
  and editor.
- `src/components/auth/auth-form.tsx` — added `WorkspaceHeader`; removed the in-card duplicate
  language toggle; compacted the mobile proof-point row.
- `src/components/onboarding/onboarding-flow.tsx` — replaced the inline header block with
  `WorkspaceHeader`.
- `src/components/editor/persistent-editor-page.tsx` — replaced `AppShell` (marketing nav +
  footer) with `WorkspaceHeader` for both the active workspace and the unavailable/setup states.
- `src/components/product-views.tsx` — tightened mobile header spacing (wordmark tracking, gaps,
  language-toggle padding) so the wordmark, language toggle, CTA, and menu button all fit at
  390px without truncation; removed the duplicated language toggle from the mobile drawer.
- `docs/visual-qa/v3-saas-design/*.png` — new screenshot set (below).
- `docs/visual-qa/v3-editor/*.png`, `docs/visual-qa/v3-onboarding/*.png` — regenerated by the
  existing Playwright visual-QA tests, now reflecting the shared header.

## Accessibility / mobile improvements

- Auth pages gained a proper landmark header with a working skip link (previously absent).
- The editor gained a skip link matching the rest of the app (previously only present on
  onboarding).
- Removed a duplicate, redundant language-toggle control from the mobile nav drawer (same control
  was already reachable in the header bar).
- Confirmed zero horizontal overflow (`scrollWidth - innerWidth === 0`) across `/`, `/auth/sign-in`,
  `/auth/sign-up`, `/onboarding` (sources/review/template steps), `/editor`, and two template
  routes at 1440×900, 1280×800, and 390×844.
- `aria-current="step"` on the onboarding progress rail, focus-to-heading on step change, and
  `prefers-reduced-motion` handling were already present and were re-verified, not re-implemented.

## Validation results

- `npm run lint` — clean, no errors or warnings.
- `npm run build` — succeeds (Next.js 16 / Turbopack production build).
- `npm run test:e2e` — 50 passed, 24 skipped, 0 failed, across the `desktop` and `mobile`
  Playwright projects. The 24 skipped tests are the pre-existing credential-gated Supabase/E2E
  tests, which remain skipped only because no Supabase project is configured in this environment
  (`.env.local` is absent; only `.env.example` exists) — consistent with prior sprints. No test
  was weakened or newly skipped by this pass. One pre-existing test (`switches between English
  and French copy`, mobile project) initially broke when the language toggle was first hidden on
  mobile to fix the crowding issue; the fix was reverted in favor of a spacing-only solution that
  keeps the toggle visible at all viewports, and the suite was re-run green.

## Backend/security areas intentionally untouched

Supabase migrations, RLS policies, storage policies, Supabase client/server auth setup, auth
server actions (`src/app/auth/actions.ts`), onboarding persistence logic, editor persistence
logic, database schema, E2E credential handling, GitHub OAuth, CV parsing, AI provider logic,
public publishing, payments, and analytics — none of these were read for modification purposes
beyond confirming their existing behavior, and none were changed.

## Remaining design debt

- The three "needs attention" style treatments (coral for blocked/rejected/invalid, amber for
  unsaved/recoverable) are semantically consistent but not documented anywhere as a formal token
  scale — worth codifying in `globals.css` as named custom properties if the palette grows further.
- `product-views.tsx` is an ~1700-line single file mixing the marketing shell, landing sections,
  demo page, templates gallery, and a legacy in-file editor demo (`EditorPage`, distinct from the
  real `PersistentEditorPage`). Splitting it by route would reduce risk on future edits but is a
  structural change beyond this design-only sprint.
- No dedicated "publish" state exists yet in the editor (correctly deferred per scope) — the
  status vocabulary (`draft` / `saving` / `saved` / `failed` / `evidence-backed` / `private`) is
  ready to extend once publishing ships.
